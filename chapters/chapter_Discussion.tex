\subsection{Knowledge Generation Mechanism}
\label{sub:knowledgeGenerationMechanism}
The first category deals with the way that \prototype{} generates its answers and how to achieve the best results when utilizing the system.

Differences in prompting style greatly influenced output characteristics. Ambiguous prompts without specialized language and terms left interpretation up to the LLM and often returned results which did not align the expectations of the experts. However, when prompts were fine tuned to resemble expert terminology and domain language, retrieval more frequently referenced elements in the knowledge graph containing the expected information. This is due to the fact that the LLM has more context to build the Cypher with. For example, when supplementing the prompt with specific key words, then the LLM has more context to base the Cypher off of. The observed results exemplify how crucial prompting is in order to achieve a desired result. In this case, it is an advantage if the user is aware of the details of the underlying knowledge graph, which allows the user to prompt specifically for the knowledge graph at hand.

\subsection{Architectural Limitations}
\label{sub:architecturalLimitations}
The test results identified several technical limitations in \prototype{}'s current version.

A frustrating moment presented itself during the final review when \prototype{} was revealed to have tricked the Cypher generation process. The Cypher queries containing a templated answer showcase how relevant the underlying system prompt is. The system prompt tasks the LLM with generating one Cypher per user request and supplements information on how to do so as well as some pointers on how to structure the Cypher. However, the system context never explicitly tells the LLM to not hardcode any self-generated information into the Cypher. This loophole within the system context was abused by the LLM. However, catching this behavior is an attest to the advantage of \prototype{} as described below in section \ref{sub:advatagesAndShortcomings}. Fixing the problem is as easy as adding a further instruction in the system prompt to prohibit this behavior.

In several interactions, Neo4j-related error messages were displayed. These errors were particularly associated with attempts by the LLM to write to the database. The current system architecture prevented permanent modifications to the database during the workshop session, as all Cyphers executed by the LLM were done in read-only mode. This design choice was deliberate, as not to allow an uncontrolled LLM to modify the knowledge graph. However, in a more mature version of \prototype{}, it may be advantageous to allow write operations directly to the database.

A further technical limitation was identified while tasking the LLM to generate an Excel file. While Open AI does support this functionality, the frontend and backend handling within \prototype{} does not. This means that the LLM may very well have generated the Excel file on the Open AI servers, but \prototype{} was not able to retrieve this file and present it to the user.

Even when no errors were being thrown, \prototype{} still displayed further technical shortcomings. One such moment was identified during the chain of prompts requesting the business support matrix. Even after multiple, refined prompts requesting a matrix, the system would not comply and instead responded with unasked for analysis. This case as well along with the above results discussing the necessary context of a prompt showcase that \prototype{} requires much guidance in order to lead it to the desired output. Often times, refined prompts were necessary in order to guide \prototype{} to an acceptable output. While this may have been acceptable during an academic review of the system, this behavioral shortcoming shows a lack of maturity for a real world system.

This can be put in contrast to Rovo's behavior, which did not throw any errors during the final review and demonstrated technical stability throughout the process. 

The above results demonstrate a level of technical immaturity within the prototype. However, within the scope of this research, a certain degree of technical immaturity was expected and taken into account. These technical shortcomings would normally be addressed during a subsequent development phase and may be improved upon during future work on \prototype{}.

\subsection{Transparency and Governance}
\label{sub:transparencyAndGovernance}
When comparing both systems, it becomes clear that \prototype{} is still in a prototypical state compared. Rovo displays technical maturity in the way it can be interacted with and how it generates its answers. However, Rovo was not unsusceptible to cases of hallucination. It can be argued that the danger of hallucinating without it being caught by the user may be much larger compared to hallucinations by \prototype{}. This is due to the lack of transparency offered by Rovo which does not allow for introspection into the generated answers.

A key advantage that \prototype{} has over commercial LLM tools such as Rovo is this aspect of transparency in its generated answers. A powerful feature of \prototype{} is that it allows the user to copy the exact Cypher used to generate a response. This enables the user to manually execute the Cypher within the same Neo4j instance and manually observe what underlying data was used for the answer. This introduced aspect of transparency is lacking in systems such as Rovo, where the user is not able to retrace how the LLM generated its answer. With this feature, the problem of hallucination can be addressed, as the user is able to fact check the generated answer with what is contained in the knowledge graph. This highlights the key difference between a blackbox system such as Rovo and a greybox system such as \prototype{}.

\subsection{Overall Goal Fulfillment}
\label{sub:overallGoalFulfillment}
With the conclusion of the final review, a retrospective comparison between the current state of \prototype{} and the expected outcome from the beginning of the research is appropriate.

Before the first development cycle commenced, it was unclear what the final system was to look like. With vague definitions of a chatbot that allows enterprise architects to interact through natural language with a proprietary knowledge graph containing their architecture data, it was unclear how this goal would be achieved. With each iterative development step during the conducted Action Research, the end goal became more and more clear, leading to the final version of the prototype.

E.g. if in archi a landscape was created with information in a "BusinessFunction" node and this is queried in Masuta, then the LLM often doesn't know that the information it should look for is in the business function node type. it searches elsewhere, doesn't find anything, and returns 0 results. This not knowing what node type to look in without supplying the context in the prompt is a necessary improvement.
	\item Document parsing with other tools may have helped speed up the process in the beginning, allowing for more well-developed features later in the project. For example doc ling https://www.docling.ai/
\end{itemize}
