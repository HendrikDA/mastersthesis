\chapter{Results and Discussion}
\label{ch:Discussion}
Now that it is clear how the study was conducted and what the architecture of the final prototype is, the results of the conducted Action Research and the state of the prototype can be categorically discussed. The results mainly reference the final review in chapter \ref{ch:finalReview}, but will also take a holistic look at the developed prototype.

\section{Knowledge Generation and Transparency Mechanisms}
\label{sub:knowledgeGenerationMechanism}
The first category deals with the way that \prototype{} generates its answers and how to achieve the best results when interacting with the system.

Differences in prompting style greatly influenced output characteristics. Ambiguous prompts without specialized language and terms left interpretation up to the LLM and often returned results which did not align the expectations of the experts. However, when prompts were fine tuned to resemble expert terminology and domain language, retrieval more frequently referenced elements in the knowledge graph containing the expected information. This is due to the fact that the LLM has more context to build the Cypher with. For example, when supplementing the prompt with specific key words, the LLM has more context to build the Cypher from. The observed results exemplify how crucial prompting is in order to achieve a desired result.

Applied to real world scenarios, this highlights two user groups who can benefit from this prompting strategy. First, it is an advantage if the user is aware of the details of the underlying knowledge graph, as this enables the user to prompt specifically for the knowledge graph at hand. If a user is aware that the knowledge graph was built off of imported Archi models and understand the schema, then they will be able to add context to the prompt to retrieve the information that they require to answer their question. A second user group who may achieve better results with the proprietary knowledge graph of the final review is employees of the examination office. Even without the necessary technical knowhow to prompt the system for keywords found in the knowledge graph, they are able to apply their own specialized terminology within the prompt. This specific context may help to generate the correct Cyphers to assist in such a use case. In contrast, a user who is neither aware of the underlying implementation nor knows the technical jargon of the examination office will struggle much more to achieve the desired result when interacting with the system.

This raises the point that a RAG-based chatbot that supports with EAM may require schema-aware interaction methods. Be it by knowledge of the underlying technology or by applying technical terms, the user's prompt greatly defines the quality of the generated output. A structured prompting strategy will return the best results when interacting with \prototype{}.

Next, the common challenge of LLMs hallucinating when generating answers could not be mitigated during this research. In the example of section \ref{sec:observedSysBehavior}, the observed reference to the QIS system, illustrates this problem. Although QIS is not mentioned in the department's architecture, it is output and described the LLM as if it were, with no indication of uncertainty. This is a clear case in which the general knowledge on which an LLM is trained is leaked into the generated answer. Hallucinations such as this may go unnoticed and although minuscule in the observed case, the effects this may have in a real world scenario could be large. The result of this is that the user must be aware that the LLM may hallucinate its answers. This requires the user to constantly fact check and ensure the correctness of outputs. When dealing with EAM data via such a RAG-based system, it is a requirement to be able to separate graph-grounded knowledge from the model's trained knowledge.

A key advantage that \prototype{} has over commercial LLM tools such as Rovo is this aspect of transparency in its generated answers. A powerful feature of \prototype{} is that it allows the user to copy the exact Cypher used to generate a response. This enables the user to manually execute the Cypher within the same Neo4j instance and observe the raw underlying data used for the answer. This introduced aspect of transparency is lacking in systems such as Rovo, where the user is not able to retrace how the LLM generated its answer. With this feature, the problem of hallucination can be addressed, as the user is able to fact check the generated answer with what is contained in the knowledge graph. This highlights the key difference between a blackbox system such as Rovo and a greybox system such as \prototype{}.

In a real world setting, this added transparency could make the difference when discussing how to implement an LLM in regard to data integrity and risk management. The enabled transparency allows for the data do be double-checked, ensuring that the data's integrity is upheld. At the same time, the risk of introducing misinformation into an enterprise's architecture may be mitigated this way.

The challenge of dealing with hallucinations goes beyond the scope of \prototype{}, as Rovo also produced factually incorrect answers and comparison between \prototype{} and Rovo brings to light the necessity for features that enable such a level of transparency. This is also confirmed in the paper by Zhao et al. (2024), where the authors mention that "enhancing transparency in LLMs and reducing the uncertainty of the generated output is critical for increasing trust and reliability their outputs, especially in fields where precision and accountability are paramount". \cite{zhao2024retrieval}

The above points were a common theme throughout the development and final review and show that having an understanding of the system at hand can give the user an advantage to extract the most practical usage from it as well as to mitigate errors.

\section{Architectural Limitations}
\label{sub:architecturalLimitations}
The results of the final review identified several technical limitations in \prototype{}'s current version.

A frustrating moment presented itself during the final review when \prototype{} was revealed to have tricked the Cypher generation process. The Cypher queries containing a templated answer showcase how relevant the underlying system prompt is. The prompt enables the LLM to complete the task, but does not ensure correctness of completion. The system prompt tasks the LLM with generating one Cypher per user request and supplements information on how to do so as well as some pointers on how to structure the Cypher. However, the system prompt never explicitly tells the LLM to not hardcode any self-generated information into the Cypher. Thus, the LLM satisfies the given instruction by generating a Cypher and using the results of the Cypher to generate the final response.

Although this loophole within the system prompt was abused by the LLM, catching this behavior is an attest to the transparency advantage of \prototype{} as described previously in section \ref{sub:knowledgeGenerationMechanism}. Fixing the problem is as simple as adding a further instruction in the system prompt to prohibit this behavior. This highlights that the system's context acts as a governance layer for the LLM to correctly fulfill the task given to it. A finely tuned system prompt is thus an important requirement for any system leaving Cypher generation up to an LLM.

In several interactions, Neo4j-related error messages were displayed. These errors were particularly associated with attempts by the LLM to write to the database. The current system architecture prevented permanent modifications to the database during the workshop session, as all Cyphers executed by the LLM were done in read-only mode. This design choice was deliberate, as not to allow an uncontrolled LLM to modify the knowledge graph. From a research perspective, this restriction was necessary to guarantee data integrity during experimentation and to ensure reproducibility of results. However, it also reveals an architectural threshold between safety and functional completeness. In a more mature version of \prototype{}, write operations could be enabled through controlled interfaces, thereby preserving governance while expanding practical applicability.

A further technical limitation was identified while tasking the LLM to generate an Excel file. While Open AI does support this file generation and output, the frontend and backend handling within \prototype{} does not currently implement the necessary mechanisms to receive, store, and transmit such artifacts. This means that even if the LLM successfully generated the file server-side, the system architecture lacked the capability to display the results to the user. The limitation is therefore not rooted in reasoning capability, but in missing infrastructural components. It highlights that the prototype primarily focuses on query generation and knowledge retrieval rather than comprehensive document management functionality.

Even when no errors were being thrown, \prototype{} still displayed further technical shortcomings. One such moment was identified during the chain of prompts requesting the business support matrix. Even after multiple, refined prompts requesting a matrix, the system would not comply and instead responded with unrequested analysis. This case highlights that an LLM does not inherently understand the operational objective behind a request, but instead optimizes for plausibility. In order to fulfill a concrete task, it must therefore be meaningfully guided through structured instructions and clearly defined expectations. While this level of orchestration was manageable during the academic review, it demonstrates that the current system requires deliberate steering to reliably produce task-oriented outputs in a real-world context.

This can be put in contrast to Rovo's behavior, which did not throw any errors during the final review and demonstrated technical stability throughout the process. The system consistently produced coherent outputs without requiring manual intervention or refined steering. This level of robustness indicates a higher degree of technical maturity and integration readiness. In comparison, \prototype{} still required active guidance and exposed instabilities that would not be acceptable in a real world environment.

The above results demonstrate a level of technical immaturity within the prototype. However, within the scope of this research, a certain degree of technical immaturity was expected and accepted. \prototype{} was conceived and evaluated as a proof-of-concept to validate architectural principles. It is not meant as a production-ready enterprise tool. Qualitative aspects such as stability represent a separate development trajectory from the core research objective of transparent knowledge graph interaction. Within that objective, functional validity was successfully demonstrated, even if technical robustness remains incomplete. These shortcomings would typically be addressed in a subsequent engineering phase and therefore constitute opportunities for future work rather than contradictions of the research contribution.

\section{Make or Buy - Comparing \prototype{} to Rovo}
\label{sub:makeOrBuy}
Although the prototype is not meant as a mature system ready to be applied in real world scenarios, it serves well as a discussion point for a typical make or buy decision when questioning whether licensing an LLM tool or building a custom tool is more advantageous for an enterprise. The argument can be framed as buying a black-box system versus developing one's own greybox system.

A further large advantage of a custom tool such as \prototype{} is that it can adopt expert knowledge via RAG. The knowledge graph containing the entirety of an EAM textbook allowed for generated answers to be grounded with expert EAM knowledge. A system such as Rovo cannot do so, as it is only able to utilize the knowledge on which it is trained along with the architectural information documented within the Confluence pages.

With vague definitions of a chatbot that allows enterprise architects to interact through natural language with a proprietary knowledge graph containing their architecture data, it was unclear how this goal would be achieved. With each iterative development step during the conducted Action Research, the end goal became more and more clear, leading to the final version of the prototype.
